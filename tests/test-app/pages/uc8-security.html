UC-8: Security Testing

This page contains intentional security vulnerabilities for automated security scanning tests. Do not deploy to production.

1. Cross-Site Scripting (XSS)

Critical

The search query from the URL parameter ?q= is reflected into the DOM using innerHTML without any escaping or sanitization. Try: ?q=<img src=x onerror=alert(1)>

Enter a search query above or pass ?q= in the URL.

2. Sensitive Data in HTML Comments

High

The page source contains credentials embedded in HTML comments. Inspect the <head> of this page to find:

<!-- API_KEY: sk-test-12345 -->
<!-- DB_PASSWORD: admin123 -->

These are readable by any user who views the page source or by automated scanners.

3. Open Redirect

High

The redirect endpoint accepts an arbitrary url parameter with no validation or allowlist. An attacker can craft phishing links that appear to originate from a trusted domain.

Click here → /redirect?url=https://evil.com Will redirect to an external URL

4. CORS Misconfiguration

Medium

Fetches /api/products and displays the Access-Control-Allow-Origin response header. A wildcard (*) or overly broad origin allows cross-origin requests from any site.

Click the button to check.

5. eval() on URL Parameter

Critical

An inline script reads the ?code= URL parameter and passes it directly to eval(). Any JavaScript supplied in the URL is executed with full page privileges. Try: ?code=alert('pwned')

Active code execution vector present on this page.
The script below runs on every page load. Passing ?code=... in the URL will execute arbitrary JavaScript. Check the DevTools console for output.
// Inline script (runs on load):
eval(new URLSearchParams(location.search).get('code'));

6. Error Information Disclosure

High

The /api/error endpoint returns a full Node.js stack trace along with internal configuration values including database credentials. This is a common misconfiguration in development environments left exposed in production.
